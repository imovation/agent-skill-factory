# 逻辑漏洞与规避指南 (Lessons Learned)

本文件记录了工厂建设过程中出现的逻辑失误及其根治方案，作为未来 AI Agent 的强制性规避指南。

## 1. 工具操作禁令 (Tool Safety)
- **现象**：使用 `Edit` 工具时意外删除了大量核心规范内容。
- **根因**：在 `oldString` 或 `newString` 参数中使用了 `...` 作为省略号，被工具识别为字面量匹配，导致非预期的替换。
- **铁律**：**严禁在 `Edit` 调用中使用 `...` 占位符。** 必须提供完整、精确且在文件中唯一的字符串块进行匹配。

## 2. 流程主权合规 (Procedural Integrity)
- **现象**：在确立“守护进化铁律”后，AI 立即违背规则，通过自动化脚本一次性修改了多个点前缀目录文件。
- **根因**：过度追求自动化（Over-automation），忽略了“意图即主权”要求的人工干预停顿。
- **铁律**：**逻辑区变更严禁“一键同步”。** 凡涉及 `.spec/` 和 `.wisdom/` 目录的修改，必须拆分为单步操作，每一轮修改后必须停顿，等待用户在 TUI 界面确认。

## 3. 上下文状态同步 (Context Management)
- **现象**：频繁触发 `File modified since last read` 报错。
- **根因**：在长对话中对文件状态的认知滞后，试图基于过期的缓存进行修改。
- **铁律**：**修改前必先读取。** 在执行任何 `Edit` 操作前，必须先执行一次 `Read` 以确保获取的是当前磁盘上的真实状态。

## 4. 交互直观性要求 (UI/UX Alignment)
- **现象**：向用户展示了难以阅读的 Git Patch 原始代码块。
- **根因**：思维局限于底层技术实现，未对齐用户在 OpenCode 环境下的视觉审查习惯。
- **铁律**：**以 OpenCode TUI 为最高审查标准。** AI 必须通过发起 `Edit` 动作来“请求”展示红绿对比视图，严禁使用非直观的文本 Diff 替代。

---
*这些教训是工厂逻辑防线的基石。任何违反上述准则的行为均视为系统级故障。*
