# 规范驱动编程 (Spec Coding) 范式

**版本**：1.2 (AI-Native Definition)
**核心哲学**：意图即主权，实现即瞬态。

---

## 一、 核心逻辑 (Core Philosophy)
在 AI 原生开发环境下，软件开发的本质从“编写语法”升维至“定义语义”：
- **文档即代码**：规范（Spec）不再是过期的注释，而是最高等级的源代码。
- **逻辑即架构**：系统的稳定性不取决于代码的堆砌，而取决于逻辑约束的严密性。

## 二、 核心要素 (The Core Elements)

### 1. 代码 (Code)：瞬态的实现 (Transient Implementation)
> **特别说明**：在 AI 原生开发时代，“代码”的含义已发生本质扩张。它不再仅仅指传统的 Java/Python 等编程语言源码，还包括供 AI 进一步处理的**提示词 (Prompts)**、OS 指令、配置文件等。人类和 AI 必须打破对“代码”一词的传统狭义理解，将其视为意图的“瞬态实现资产”。

- **本质**：代码是意图在特定技术栈下的“折叠”和“投影”，是可丢弃、可再生的中间产物。
- **定义**：基于规范生成的、供计算机执行的指令集或供 AI 进一步处理的提示词。
- **交付物示例**：Java/Python 源码、AI 提示词 (Prompts)、AgentSkill 逻辑、OS 指令、配置文件、说明文档。
- **核心特性**：
    - **AI 所有权**：AI 是代码编写与逻辑验证的第一负责人。
    - **环境受限**：代码的正确性必须由测试环境闭环验证，无法自动验证的部分必须请求人类协助。
    - **单向投影**：从 Spec 到 Code 的转换必须是自动、幂等且单向的。

### 2. 规范 (Spec)：结构化的意图 (Structured Intent)
- **本质**：规范是人类意志的逻辑化表达，是系统的“单一事实来源（SSOT）”。
- **定义**：由人类编写或由 AI 编写并经人类确认的，用于规约 AI 编程方向的约束性文档。
- **特性**：
    - **人类主权**：人类是规范的唯一最终决策者。AI 负责提供提案，人类负责定案。
    - **溯源一致性**：严禁在代码层进行手动修改。任何变更必须回溯至 Spec。

## 三、 双轮驱动架构 (Dual-Wheel Drive)

### 1. 规范驱动 (Spec-Driven)：做正确的事 (向上对齐)
- **原则**：先规范，后代码。严禁在没有规范定义的情况下开始编码。
- **同步性**：任何代码的变更必须追溯至规范的更新，始终保持规范与代码的同频。

### 2. 验证驱动 (Validation-Driven)：把事做对 (向下渗透)
- **原则**：在实现逻辑之前，先定义验证逻辑（状态探针）。
- **流程**：遵循“探针红（预期失败） -> 探针绿（实现成功） -> 逻辑固化”的闭环。

## 四、 协作流程协议 (The Protocol)

| 角色 | 负责领域 | 核心职责 |
| :--- | :--- | :--- |
| **人类 (Human)** | **规范 (Spec)** | 定义边界、决策风险、确认最终逻辑 |
| **AI (Agent)** | **编程 (Coding)** | 编写代码、执行验证、提出规范优化建议 |

### 开发流水线：
1. **意图输入**：人类定义初始规范。
2. **规范定稿**：人类审查并冻结 Spec 语义。
3. **代码投影**：AI 基于 Spec 全自动生成实现代码。
4. **闭环验证**：AI 执行测试。若失败则自迭代；若受环境限制无法验证，则上报人类协助。

## 五、 绝对红线 (Red Lines)
1. **篡改投影**：严禁在生成的代码投影层手动打补丁，绕过 Spec 的修改是系统腐化的开始。
2. **模糊语义**：Spec 必须是高密度的逻辑约束，严禁出现导致 AI 产生幻觉的模糊描述。
3. **脱离上下文修改**：严禁 AI 在未完全理解 Spec 全局语义的情况下进行局部代码重构。

## 六、 反馈闭环与自进化 (Feedback Loop)
- **运行时反馈**：收集运行日志与用户反馈，自动评估代码效能。
- **规范建议**：当 AI 发现规范存在逻辑冲突或执行效率低下时，应主动向人类发起修改提案。
- **持续优化 (Möbius Loop)**：通过“反馈 -> 修正规范 -> 重新投影代码”实现系统的无限演进。

---
**结语**：在 Spec Coding 的世界里，开发者不再是代码的搬运工，而是逻辑的立法者。
